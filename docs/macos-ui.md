# ZSUI macOS Host Plan

This plan tracks the macOS native host for ZSUI, the shared Rust UI contract
being extracted from ZSClip.

## Current State

- `src/main.rs` now has separate Windows and macOS entry points.
- Windows keeps the existing Win32 host behind `target_os = "windows"`.
- macOS starts at `src/macos_app.rs` behind `target_os = "macos"`.
- The macOS host currently reads the ZSUI `app_core` contract summary; it does not copy Windows `app.rs`.
- `src/macos_appkit_adapter.rs` now defines the AppKit/SwiftUI adapter boundary, required host binding names and settings widget roles for the future native implementation.
- Required host surfaces now live in `app_core::ui_surface_protocol`; macOS consumes that shared list for main window, settings window, dropdown and dialog adapters.
- The macOS scaffold now also tracks the shared clipboard host contract.
- The contract summary tracks `StatusItemHost` operations needed by the native menu bar item.
- `MacosStatusItemHost` now consumes shared status-menu entries and lifecycle calls; the first real AppKit host slice presents those shared entries through an `NSStatusItem` menu, with target macOS smoke evidence still required.
- `MacosPopupMenuHost` now consumes shared popup-menu entries and the macOS scaffold verifies shared row/group menu plans; the first real AppKit host slice maps full Row Menu entries and Group Filter entries to `NSMenu`, and the main host can open a first-pass VV Popup native window from the shared render plan with row selections bridged through `vv_select_requested`. Real VV trigger integration, native paste execution and target smoke evidence remain pending.
- `MacosDialogHost` now tracks shared message and confirmation dialogs, and the real AppKit settings surface exposes first-pass `NSAlert` Info/Confirm buttons through shared native dialog actions.
- `MacosMainWindowHost` now consumes `NativeMainWindowRequest`, returns main/quick window handles, owns platform appearance application and records native activation-policy, pointer-capture, pointer-leave tracking, repaint, layout-DPI, client/window-bounds, native window-drag, hide/activate/foreground/restore/close/destroy requests, while `MacosWindowSessionState` records the main host appearance, bounds and activation-policy generation, so startup window creation, native window chrome, geometry, refresh, drag tracking and basic window lifecycle have a shared host boundary.
- `MacosWindowSessionState` now also records edge auto-hide enabled/hidden state, bounds and generation; future AppKit edge docking should use this Rust session shape while replacing only the native screen-edge detection and animation calls.
- `MacosMainWindowModel::startup_plan` now builds the native main-window request and mount lifecycle event from the shared layout; macOS `run()` executes that plan through `MacosMainWindowHost` and applies native appearance to the created main window.
- `MacosApplicationModel` now owns the shared lifecycle state, command queue, main/settings window models and `MacosMainEventModel`; macOS `run()` mounts and activates this persistent Rust state root instead of assembling unrelated temporary models.

## Direction

- Keep `app_core`, settings/sync models and database models shared.
- Implement macOS as a native host using AppKit or SwiftUI concepts.
- Do not port Win32 messages, `HWND`, GDI drawing helpers or Windows control class names to macOS.
- Every macOS feature should consume the same plans and contracts that Windows consumes.
- ZSUI shared framework code defines behavior, state, layout plans and component contracts. It does not force Windows and macOS to share one visual skin.

## First MVP

1. App starts through the macOS entry point.
2. Add a native menu bar / status item.
3. Wire real VV trigger integration and native paste execution to the VV Popup select bridge, then capture target macOS smoke evidence.
4. Verify and expand native `NSAlert` presentation for shared simple dialogs on a real macOS runner.
5. Open a native clipboard history window and execute the existing shared main-window render plan.
6. Render copied text records from shared models.
7. Extend the `NSPasteboard` host from file URL read/write and `changeCount` into continuous monitoring, source filtering and target smoke coverage.

- macOS main windows need `NativeMainWindowHost`, so Windows main/quick class registration, creation, native appearance, activation policy, present strategy, bounds positioning/query, layout-DPI query, repaint, pointer-leave tracking, pointer capture, native window dragging and basic hide/activate/foreground/restore/close/destroy lifecycle now route through `WindowsMainWindowHost` in `src/app/main_window_host.rs`.

Complete settings forms, target StatusNotifier proof, real VV trigger integration, successful OS paste shortcut proof under target permissions, clipboard monitoring and platform identity services remain target-OS work.

- The product AI layer now exposes `product_ai_integration_manifest()`, so AppKit/SwiftUI, GTK/libadwaita and Win32 hosts can inspect LLM, skill and product-adapter provider/action routes without hard-coding provider ids or menu-specific action strings.

- The important architectural constraint is the same as macOS: Linux code should implement native hosts and platform bindings, not copy Windows `app.rs` message handling.
